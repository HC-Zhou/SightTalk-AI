## 2. 产品定位

SightTalk AI 是一个 AI 视觉对话助手。用户通过浏览器授权摄像头和麦克风后，可以像视频通话一样与 AI 交流；AI 能听懂用户语音、理解当前画面，并用自然语音和文本回应。

MVP 的核心价值是让用户在现实场景中“指给 AI 看，再直接问”。产品应优先保证画面相关回答可信、交互链路顺畅、隐私边界清楚，而不是追求复杂 Agent 编排或长期记忆。

## 3. 目标用户与典型场景

| 用户类型 | 场景 | 价值 |
| --- | --- | --- |
| 日常生活用户 | 看菜单、包装、说明书、路牌、场景布置 | 快速获得解释、建议和下一步动作 |
| 学习和办公用户 | 对准文档、白板、屏幕内容追问 | 把视觉信息转成可理解的说明 |
| 轻量决策用户 | 对商品、菜品、环境进行比较 | 获得基于画面证据的建议 |

首个验收场景选择“菜单减脂推荐”：用户将摄像头对准菜单，询问“我想减脂，点什么比较合适”，随后追问“如果我不吃辣呢？”或“这两个哪个更好？”。

## 4. 产品目标

### 4.1 计划实现

SightTalk AI 的完整产品方向包括：

- 实时视频对话：用户能用摄像头、麦克风与 AI 连续交流。
- 视觉问答：AI 基于当前画面回答菜单、实物、文档、环境类问题。
- 多轮追问：AI 能记住最近对话中的视觉摘要和用户偏好。
- 语音自然交互：AI 以语音播报，用户可以打断并进入新问题。
- 隐私控制：用户能随时暂停画面上传，产品明确说明当前是否上传画面。
- 弱网和低质量画面处理：AI 不编造，提示用户靠近、重拍或补充说明。
- 成本可控：默认使用低成本模型和关键帧策略，仅在必要时升级模型。

### 4.2 MVP 实现范围

MVP 只交付能闭环验证核心价值的最小产品：

- 浏览器端摄像头和麦克风授权、视频预览、连接状态展示。
- 基于 LiveKit 的实时音频、视频和数据通道。
- 用户语音识别、当前关键帧抽样、视觉问答、TTS 播报。
- 最近 10-20 轮短期上下文和视觉摘要，用于追问。
- 隐私模式开关：开启后停止上传关键帧，AI 必须知道当前不可见画面。
- TTS 可被新一轮用户语音打断。
- 对模糊、遮挡、过远、弱网等情况给出澄清或重拍提示。
- 原始音频、视频、图片不落盘，仅保留必要的短期内存态。

### 4.3 MVP 不做

- 多 Agent 协作、复杂工作流编排。
- 长期记忆、账号体系、跨设备历史记录。
- 全量本地 OCR、目标检测、图像分割模型落地。
- 移动端原生 App。
- 社交分享、社区、订阅付费、复杂后台管理。
- 对医疗、法律、金融等高风险问题给出确定性专业结论。

## 5. 用户故事

| ID | 用户故事 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| US-01 | 作为用户，我可以授权摄像头和麦克风，进入视频对话。 | P0 | 首次进入页面时浏览器弹出授权；授权成功后能看到本地视频预览和连接状态；拒绝授权时显示可恢复的错误提示。 |
| US-02 | 作为用户，我可以对准实物、菜单、文档或场景提问。 | P0 | 用户说出问题后，系统能采集语音和当前关键帧，并返回与画面相关的回答。 |
| US-03 | 作为用户，我希望 AI 回答时引用画面证据，而不是泛泛而谈。 | P0 | 回答中必须包含“我看到/画面中/菜单上”等视觉依据；无法看清时必须说明不确定。 |
| US-04 | 作为用户，我可以追问上一轮视觉内容。 | P0 | 用户追问“那这个呢”“刚才你说的第二个呢”时，AI 能结合最近视觉摘要和对话上下文回答。 |
| US-05 | 作为用户，我可以打断 AI 播报并进入新一轮对话。 | P0 | AI 播报中用户开始说话时，当前 TTS 停止或淡出，新语音进入识别和回答流程。 |
| US-06 | 作为用户，我可以开启隐私模式，停止上传画面。 | P0 | 隐私模式开启后不再发送关键帧；AI 回答时明确当前看不到新画面；关闭后恢复画面理解。 |
| US-07 | 作为用户，在弱网或画面不清晰时，我希望 AI 提示我如何补救。 | P1 | 模糊、过暗、遮挡、过远时，AI 不编造内容，并提示重拍、靠近、补光或补充文字描述。 |
| US-08 | 作为用户，我可以看到实时字幕和回答文本。 | P1 | 页面展示用户转写文本、AI 回答文本和基本会话状态，刷新后不要求保留。 |

## 6. 核心交互流程

### 6.1 首轮视觉问答

1. 用户打开页面并授权摄像头、麦克风。
2. 前端加入 LiveKit 房间，展示本地视频预览。
3. 用户对准菜单并说：“我想减脂，点什么比较合适？”
4. Agent 获取语音转写和当前关键帧。
5. VLM 根据菜单画面生成建议，回答时引用可见菜名、烹饪方式或图片特征。
6. 前端展示回答文本，LiveKit 播放 Agent 发布的 TTS 音频。

### 6.2 追问

1. 用户继续问：“如果我不吃辣呢？”
2. Agent 复用上一轮视觉摘要和当前关键帧。
3. AI 将回答范围限定在菜单可见内容和用户限制条件内。

### 6.3 打断

1. AI 正在播报时，用户开始说新问题。
2. 前端或 Agent 侧检测到用户语音活动。
3. Agent 停止当前 TTS 输出，并进入新一轮识别。

### 6.4 隐私模式

1. 用户开启隐私模式。
2. 前端停止关键帧采样和发送。
3. AI 可以继续语音对话，但不能声称看到新画面。
4. 用户关闭隐私模式后，前端恢复关键帧采样。

## 7. 功能需求

### 7.1 前端

- 摄像头、麦克风权限申请和异常提示。
- 本地视频预览、连接状态、麦克风状态、AI 播报状态。
- 隐私模式开关，状态必须清晰可见。
- 实时字幕和回答文本展示。
- 关键帧采样、压缩、去重和 Data Channel 发送。
- 用户打断入口：检测用户开始说话时通知 Agent 或直接触发停止播报。

### 7.2 后端 API

- 签发 LiveKit 房间 Token。
- 管理会话基础配置，例如模型档位、关键帧频率、超时策略。
- 暴露健康检查和配置检查接口。
- 不接收、不持久化原始音视频文件。

### 7.3 Agent Worker

- 订阅用户音频轨道，调用 ASR。
- 接收前端关键帧，做质量检查、压缩后的去重和节流。
- 将用户意图、转写文本、关键帧、视觉摘要传入 VLM。
- 调用 TTS 并发布音频回 LiveKit。
- 维护最近 10-20 轮短期上下文和视觉摘要。
- 处理用户打断、会话空闲超时和错误恢复。

## 8. 非功能需求

| 类型 | 要求 |
| --- | --- |
| 隐私 | 原始音频、视频、图片不落盘；日志不记录原始图片、完整音频或敏感凭证。 |
| 性能 | 首轮回答链路应以实时对话体验为目标；关键帧数量、图片尺寸和模型档位必须可配置。 |
| 可靠性 | 弱网、模型超时、权限失败时提供可恢复提示。 |
| 可维护性 | 前端、后端 API、Agent Worker 独立演进；模型供应商通过适配层隔离。 |
| 成本 | 默认使用 Flash 档模型；Plus 档升级必须由策略触发并可观测。 |

## 9. 验收标准

MVP 达到以下标准才算可验收：

- 菜单减脂推荐场景可完成首轮回答和至少一轮追问。
- 普通物体、菜单、文档或场景问答必须引用画面证据。
- 模糊、遮挡、过暗或画面不可见时，AI 不编造具体内容。
- TTS 播报可被用户新语音打断，并进入新一轮对话。
- 隐私模式开启后不上传新关键帧，AI 不声称看到了新画面。
- 原始音频、视频、图片不落盘；本地开发日志中不出现原始媒体内容。
- LiveKit Token 有过期时间，前端不能直接持有 LiveKit API Secret 或 DashScope API Key。

## 10. 风险与开放问题

| 风险 | 影响 | MVP 应对 |
| --- | --- | --- |
| 模型视觉误判 | 回答可信度下降 | 强制引用画面证据；不确定时提示重拍或澄清。 |
| 端到端延迟过高 | 对话不自然 | 关键帧节流、Flash 默认、TTS 可打断。 |
| 图片上传成本失控 | 个人项目成本不可控 | 压缩、hash 去重、空闲超时、Plus 受控升级。 |
| 浏览器权限和设备差异 | 首次使用失败 | 明确权限错误和设备选择提示。 |
| DashScope 模型和地域限制变化 | 上线集成失败 | 上线前按 DashScope 控制台和官方文档复核模型名、地域和计费。 |

待后续确认的问题：

- 是否需要登录后保存非原始媒体的会话摘要。
- 是否支持用户手动截图提问，而不只依赖自动关键帧。
- Plus 档升级是否需要用户显式确认，还是仅用成本策略控制。
- 是否需要为菜单、文档、物体识别分别设计专门提示词模板。
